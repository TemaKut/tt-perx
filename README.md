# tt-perx

# Коммент к реализации
Выполнил в виде микросервиса с архитектурным паттерном SOA 
DI - wire. Все "грязные" операции по формированию зависимостей находятся в ./cmd/factory оставляя ./internal более чистым
Cli - для передачи аргументов cmd используется библиотека "github.com/urfave/cli/v2"

Сервис имеет в себе 3 основных слоя:
 - Хендлеры -> для взаимодействия с клиентами и тд
 - Сервисный слой - основная бизнес-логика
 - Сторадж - слой хранения данных

Для коммуникации между хендлерами и сервисным слоем используется DTO, далее сервисный слой использует бизнес-модели


Что хорошо сделать для продакшна:
    - Навешать метрики (Prometheus + Grafana)
    - Вывести логирование (Graylog)
    - Сделать персистентное хранилище (Скорее реляционная БД по типу PostgreSQL) для того чтобы я мог реплицировать сервис
    - Для репликации сервиса можно закинуть его в K8S или Nginx
    - Добаить тестов как интеграционных, так и модульных
    - Ci\cd
    - Обернуть в Docker с мульти образной сборкой для экономии памяти
    - Брать переменные окружения из .env
